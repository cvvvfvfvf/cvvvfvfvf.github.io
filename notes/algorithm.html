Algorithm Design & Analysis Notes

Algorithm Design and Analysis · Fall 2025

Algorithms

Definition

A description of the steps for solving a specific problem; a finite sequence of instructions.

Five Characteristics

  • Input: Algorithms have zero or more inputs

  • Output: Algorithms produce at least one output

  • Definiteness: Each instruction must be precisely defined with no ambiguity

  • Effectiveness: Each instruction must be basic enough to be executed finitely using already implementable basic operations

  • Finiteness: Algorithms must always terminate after a finite number of steps

Programs

  • Definition: A concrete implementation of an algorithm using a programming language

  • Operating System: A program that executes in an infinite loop, therefore NOT an algorithm

  • Various OS tasks can be viewed as individual problems, each solved by specific algorithms within subroutines that terminate after producing outputs

Why Study Algorithms?

  • Algorithms are the foundation of computer science

  • They form the bedrock of programming

  • A solid grasp of algorithms is essential for becoming a well-trained software professional

  • The purpose of algorithms is to solve problems

Six Common Application Problem Types

Search problems, sorting problems, graph problems, combinatorial problems, geometric problems, numerical computation problems

Problem Solving

Definition: Finding a method to achieve a goal. A key aspect of computer-based problem solving is finding a problem-solving strategy that yields an algorithm, which then produces the solution.

Nine Steps of the Problem-Solving Process

  1. Understand the problem

  2. Enumerate all possible inputs

  3. Choose between exact and approximate solutions

  4. Determine appropriate data structures

  5. Algorithm design techniques: greedy, divide-and-conquer, dynamic programming, etc.

  6. Describe the algorithm

  7. Trace the algorithm

  8. Analyze algorithm efficiency

  9. Write code based on the algorithm

Four Algorithm Classifications

  • Exact algorithms: Always guarantee finding a solution

  • Heuristic algorithms: Reduce solution time using rules, simplifications, or intelligent guesses

  • Approximation algorithms: For optimization problems, algorithms that seek approximate rather than optimal solutions

  • Randomized algorithms: Algorithms that make random choices during execution

Algorithm Design

Using computer-oriented problem-solving strategies (algorithm design strategies)

Four Algorithm Description Methods

Natural language, flowcharts, pseudocode, programming languages

Four Ways to Validate Algorithms

  • Proof of Correctness: Using mathematical methods to prove correctness

  • Program testing: Running pre-prepared sample data (test cases) to discover errors and determine if the program meets design requirements

  • Algorithm analysis: Estimating execution time and space requirements

  • Performance measurement: After implementation, measuring actual time and space consumption using sample data

Program Performance

Memory space and computation time required during program execution

Two Performance Evaluation Methods

Analytical methods, measurement methods

Algorithm Complexity

Time complexity, Space complexity. Depends only on problem size, input, and the algorithm itself.

Space Complexity ($S_p(n)$)

Space requirements (ignoring instance-independent factors): the relationship between memory space needed during program execution and problem instance characteristics.

  • Instruction space: instance-independent constants

  • Data space: constants and simple variables (instance-independent), composite variables (arrays, linked lists, trees, graphs), environment stack space (function calls) — recursive or not?

  • Composite variable space requirements often depend on problem instance characteristics

Master Method Analysis

  • \(n=b^h, h=\log_b n, n^{\log_b a}=a^h\)

  • \(\log_b a\) denotes the logarithm of a base b

  • \(T(n)=a^h T(1)+\sum_{k} a^k f(n/b^k)\) \[ =a^h T(1)+\sum_{k} a^k f(b^{h-k}) \]

    \(\sum_k\) summation range: \(k=0,...,h-1\)

  • \(\sum_k a^k f(b^{h-k})=\sum_k a^{h-k} f(b^k)\), (the latter sums from 1 to h)

Proof (Case 1)

  • $n = b^h$, then $n^{\log_b a} = a^h$, $n^\epsilon = b^{\epsilon h}$. \(f(n) = O(n^{\log_b a - \epsilon})\), for sufficiently large \(c\), $ f(n) \leq c n^{\log_b a - \epsilon} = c a^h / b^{\epsilon h} \Rightarrow f(b^h) \leq c (a^h / b^{\epsilon h}) $

  • $T(n) = a^h T(1) + \sum a^k f(b^{h-k})(\sum : k = 0, \ldots, h-1) \leq a^h T(1) + c \sum a^k (a^{h-k} / b^{\epsilon (h-k)}) = a^h T(1) + c a^h \sum (1 / b^{\epsilon (h-k)}) \leq a^h T(1) + c a^h \sum_{1 \leq k < \infty} (1 / b^{\epsilon k}) \leq C a^h \quad (b > 1, \epsilon > 0$ the infinite sum converges)

Proof of Case 2

  • $f(b^h) \leq c a^h (\ln b^h)^k$

  • \(T(n) \leq a^h T(1) + c a^h \sum_i j^k (\ln b)^k \leq = a^h T(1) + c a^h \ln^{k+1} (\lg b)^{k+1} / \lg b = c'' (n^{\log_b a} \lg^{k+1} n)\)

  • The above derivation uses: $ 1^k + \ldots + n^k = \Theta(\ln^{k+1})$

Proof of Case 3

  • $ a^k f(n/b^k) \leq c^k f(n)\Rightarrow\sum a^k f(n/b^k) \leq \sum c^k f(n)$

  • \(T(n) = a^n T(1) + \sum a^k f(n/b^k) \leq a^n T(1) + f(n) \sum c^k \leq a^n T(1) + f(n)(1-c)^{-1}\)

  • Because \(f(n) = \Omega(n^{\log_b a + \epsilon})\) has a higher magnitude than \(a^n\), we have $T(n) = O(f(n))$

  • Also, \(T(n) > f(n)\), therefore \(T(n) = \Theta(f(n))\)

Recursive Function Example

void recursive(int n, int m, int o) {
	if (n <= 0) {
		printf("%d, %d\n", m, o);
	} else {
		recursive(n - 1, m + 1, o);   // Recursive call 1
		recursive(n - 1, m, o + 1);   // Recursive call 2
	}
}

The function termination condition is $n \geq 0$. Although parameters $m$ and $o$ change during recursion, they do not affect the recursion depth or branching factor. Each call produces two recursive calls, and $n$ decreases by 1 each time, so the time complexity satisfies:

\[ T(n) = 2T(n-1) = O(2^n) \]

The function termination condition is \(n \geq 0\), followed by two recursive calls to itself. Since parameters \(m\) and \(o\) do not participate in the termination condition, the running time simplifies to \(T(n) = T(n - 1) + T(n - 1)\), i.e., \(T(n) = 2 \times T(n - 1)\). This is a recurrence relation:

  • \(T(n) = 2 \times T(n - 1)\)

  • \(T(n - 1) = 2 \times 2 \times T(n - 2)\)

  • \(T(n - 2) = 2 \times 2 \times 2 \times T(n - 3)\)

  • ...

Thus the complexity is \(O(2^n)\).

Divide-and-Conquer Algorithms

Applicable Conditions

  • The problem can be easily solved when reduced to a sufficiently small size

  • The problem can be decomposed into several smaller subproblems of the same type

  • The solutions of the subproblems can be combined into the solution of the original problem

  • The subproblems are independent of each other, i.e., they do not share common subproblems

Divide-and-Conquer Algorithm Framework

Divide-and-Conquer Framework
Input: Problem instance $P$
Output: Problem solution
if $|P| \leq n_0$ then
$S(P)$ // Solve small-scale problem
Decompose $P$ into smaller sub-instances $P_1, P_2, \ldots, P_k$ // Decompose the problem
for $i = 1$ to $k$ do
$y_i \gets \text{divide-and-conquer}(P_i)$ // Recursively solve each subproblem
return $\text{merge}(y_1, \ldots, y_k)$ // Combine subproblem solutions into original problem solution

Merge Sort Implementation

void MergeSort(Array A, int left, int right) {
	if (left >= right) return;
	int mid = (left + right) / 2;
	MergeSort(A, left, mid);
	MergeSort(A, mid + 1, right);
	Array temp = new Array[right - left + 1];
	int i = left, j = mid + 1, k = 0;
	while (i <= mid && j <= right) {
		if (A[i] <= A[j])
		temp[k++] = A[i++];
		else
		temp[k++] = A[j++];
	}
	while (i <= mid) temp[k++] = A[i++];
	while (j <= right) temp[k++] = A[j++];
	for (int p = 0; p < k; p++)
	A[left + p] = temp[p];
	delete[] temp;
}

  • Given an integer \(r\), divide the \(n\) elements of array \(a\) into \(\lceil n/r \rceil\) groups, each with \(r\) elements. Any remaining elements (less than \(r\)) are not candidates for pivot selection.

  • Recursively apply the selection algorithm to select the median of the medians as the pivot element

Selection Algorithm

\[ \Theta(n) \quad \begin{cases} 1. \text{Divide the } n \text{ elements into groups of } 5. \text{ Find the median of each } 5\text{-element group by rote}. \\ 2. \text{Recursively SELECT the median } x \text{ of the } \lfloor n/5 \rfloor \text{ group medians to be the pivot}. \\ 3. \text{Partition around the pivot } x. \text{ Let } k = \text{rank}(x). \end{cases} \] \[ \Theta(n) \quad \begin{cases} 4. \text{if } i = k \text{ then return } x \\ \text{else if } i < k \\ \text{then recursively SELECT the } i\text{th smallest element in the lower part} \\ \text{else recursively SELECT the } (i-k)\text{th smallest element in the upper part} \end{cases} \]

When selecting the pivot using the "median of medians" rule, the following holds: if \(r=5\) and all elements in \(a\) are distinct, then for \(n \geq 24\), \(\max\{| \text{left} |, | \text{right} |\} \leq 3n/4\). Therefore, each recursive call reduces the problem size by at least $1/4$, giving the recurrence:

\[ T(n) \leq T(3n/4) + T(n/5) + O(n) \]

which solves to $T(n) = O(n)$.

Proof

According to the median of medians rule, at least \(\left\lceil \frac{3}{2} \right\rceil \frac{n}{5}\) median elements are not greater than the pivot (at least $n/5/2$ groups have their median $\le$ pivot, with 3 elements per group $\le$ pivot); the number of elements greater than the pivot has an upper bound

\[ n - \frac{3}{2} \left\lfloor \frac{n}{5} \right\rfloor \leq n - 1.5(n/5 - 4/5) = 0.7n + 1.2 \]

Therefore, when \(n \geq 24\), \(0.7n + 1.2 \leq 3n/4\)

Flow Shop Scheduling Problem

\[ T(S, t) = \min_{i \in S} \left\{ a_i + T\left(S \setminus \{i\},\; b_i + \max\{t - a_i, 0\}\right) \right\} \]

  • $S$: Set of jobs to be scheduled

  • $t$: Time offset when machine $M_2$ can start processing

  • $a_i$: Processing time of job $i$ on machine $M_1$; $b_i$: Processing time of job $i$ on machine $M_2$

Johnson's Inequality

Definition: Jobs $i$ and $j$ satisfy $\min\{b_i, a_j\} \geq \min\{b_j, a_i\}$, then we say jobs $i$ and $j$ satisfy Johnson's Inequality.

  • Analysis of the first two jobs in an optimal schedule
    Let the first two jobs in an optimal schedule be $i$ and $j$: \begin{align*} T(S, t) &= a_i + T\bigl(S \setminus \{i\},\; b_i + \max\{t - a_i, 0\}\bigr) \\ &= a_i + a_j + T\bigl(S \setminus \{i, j\},\; t_{ij}\bigr) \end{align*} \begin{align*} t_{ij} &= b_j + \max\{b_i + \max\{t - a_i, 0\} - a_j,\; 0\} \\ &= b_j + b_i - a_j + \max\{\max\{t - a_i, 0\},\; a_j - b_i\} \\ &= b_j + b_i - a_j + \max\{t - a_i,\; a_j - b_i,\; 0\} \\ &= b_j + b_i - a_j - a_i + \max\{t,\; a_i + a_j - b_i,\; a_i\} \end{align*}

  • Effect of swapping job order
    After swapping the order of jobs $i$ and $j$: $t_{ji} = b_j + b_i - a_j - a_i + \max\{t,\; a_i + a_j - b_j,\; a_i\}$

  • Equivalent form If Johnson's inequality holds: \begin{align*} \min\{b_i, a_j\} &\geq \min\{b_j, a_i\} \\ -\max\{-b_i, -a_j\} &\geq -\max\{-b_j, -a_i\} \quad \text{(multiply by -1 and reverse)}\\ \max\{-b_i, -a_j\} &\leq \max\{-b_j, -a_i\} \\ a_i + a_j + \max\{-b_i, -a_j\} &\leq a_i + a_j + \max\{-b_j, -a_i\} \\ \max\{a_i + a_j - b_i,\; a_i\} &\leq \max\{a_i + a_j - b_j,\; a_i\} \end{align*} \[ \max\{t,\; a_i + a_j - b_i,\; a_i\} \leq \max\{t,\; a_i + a_j - b_j,\; a_i\} \]
  • Conclusion \[ t_{ij} = b_j + b_i - a_j - a_i + \max\{t,\; a_i + a_j - b_i,\; a_i\} \] \[ t_{ji} = b_j + b_i - a_j - a_i + \max\{t,\; a_i + a_j - b_j,\; a_i\} \] and $\max\{t,\; a_i + a_j - b_i,\; a_i\} \leq \max\{t,\; a_i + a_j - b_j,\; a_i\}$, so $t_{ij} \leq t_{ji}$

    This means: If jobs $i$ and $j$ satisfy Johnson's inequality, then scheduling $i$ before $j$ is no worse than scheduling $j$ before $i$.

Johnson's Scheduling Rule

  1. Divide jobs into two categories:

    • Jobs with $a_i \leq b_i$ (Class U)

    • Jobs with $a_i > b_i$ (Class V)

  2. Sort Class U jobs in increasing order of $a_i$

  3. Sort Class V jobs in decreasing order of $b_i$

  4. Schedule Class U jobs first, then Class V jobs in order

This rule ensures that adjacent jobs satisfy Johnson's inequality, thus achieving a globally optimal schedule.
